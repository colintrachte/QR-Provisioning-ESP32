 PROJECT VISION
═══════════════════════════════════════════════════════════════

Inspired by Meshtastic — but built for robotics, not messaging.
Goal: a self-contained ESP-IDF foundation that handles the hard
infrastructure (WiFi provisioning, OTA, captive portal, health
monitoring, display, WebSocket control) so that new use cases
can be bootstrapped quickly by swapping a single frontend template
and implementing a thin application task.

No cloud required. No subscriptions. Optional cloud integrations
are fine as an add-on.

Target use cases (all share the same core):
  - Realtime RC control (tank, rover, arm)
  - Device-to-device control (direct or through a gateway)
  - Sequenced command playback (pre-arranged move lists)
  - Autonomous with periodic status push (e.g. chicken coop door → Home Assistant)
  - Smart home platform integration (Home Assistant, Google, Amazon)
  - Custom Unity / game-engine app as frontend

 OPEN — Firmware / Backend
───────────────────────────────────────────────────────────────

[ ] Wire motor_drive(x, y), motor_stop(), motor_set_led(val) stubs
    in wifi_manager.c to an actual motor driver component.

[ ] Push telemetry from firmware over WebSocket.
    Suggested JSON shape (matches what script.js already parses):
      { "left":  { "speed": 0, "state": "STOPPED", "forward": true },
        "right": { ... },
        "rssi": -55, "battery": 87, "temp": 32.1,
        "uptime": 120, "heap": 180000, "cpu": 12, "errors": 0 }
    Send from app_task at whatever rate makes sense (suggest 5–10 Hz).

[ ] OTA firmware update — detect available version, download and
    flash via esp_https_ota. Preserve NVS user settings. Warn when
    settings cannot survive the update. Also support OTA for LittleFS
    (web file update without full reflash).

[ ] Dynamic user settings in LittleFS — a JSON file for runtime
    config (calibration, selected UI template, LoRa region, etc.)
    that survives firmware updates. Expose a simple GET/POST API.

[ ] UI template switching — store selected template name in user
    settings. App server serves the matching set of HTML/CSS/JS.
    Current "RC tank" UI is template #1.

[ ] Peripheral config system — at boot, read hardware profile from
    user settings (which sensors are present, which GPIOs they use).
    health_monitor already scans I2C; extend it to act on findings.

───────────────────────────────────────────────────────────────
 OPEN — Comms
───────────────────────────────────────────────────────────────

[ ] Evaluate WebSocket vs UDP for realtime RC control.
    WebSocket: reliable, easy, browser-native, ~2–5 ms overhead on LAN.
    UDP: lower latency, no retransmit, needs a native app or custom
    browser shim. Recommend prototyping UDP in parallel and comparing
    measured latency + jitter under load before committing.

[ ] BLE provisioning path — alternative to SoftAP for phones that
    handle BLE pairing more gracefully than captive portals.

[ ] LoRa integration — join existing Meshtastic network OR run a
    custom point-to-point protocol. Must enforce regional duty-cycle
    limits (EU 1%, US unrestricted ISM, etc.). See Meshtastic region
    table for reference. Requires separate HAL per radio module.

[ ] Home Assistant / MQTT integration — publish sensor data and
    accept command topics. Store broker address in user settings.

[ ] Device-to-device control — one ESP32 acts as controller,
    another as actuator, communicating over WiFi (UDP/WS) or LoRa.

───────────────────────────────────────────────────────────────
 ARCHITECTURE NOTES (decisions made, rationale preserved)
───────────────────────────────────────────────────────────────

File system:    LittleFS only. SPIFFS is deprecated.
Framework:      ESP-IDF 5.x + PlatformIO as developer shell only.
                PlatformIO library manager is NOT used for components.
Libraries:      Vendored as git submodules in components/ except
                u8g2_hal which is downloaded manually due to S3 patches.
Config split:   sdkconfig.defaults → silicon/driver settings.
                platformio.ini → build environment metadata.
                config.h → application constants (baked in at compile time).
                LittleFS JSON → runtime user settings (survives OTA).
Code style:     Allman braces, concise comments, @param on every function.
                Max ~3 files per AI session for reliable code generation.
UI philosophy:  All actions independent of display/web UI.
                Motor runs if OLED dead. Motor runs if browser disconnected.
                UI is telemetry + convenience, not a dependency.
